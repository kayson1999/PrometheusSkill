## 📖 介绍

### 这个 Skill 能干什么？

Prometheus Skill 是一个面向 AI Agent 的**通用监控操作技能**，覆盖 Prometheus 监控体系的核心操作场景：

| 能力 | 说明 |
|------|------|
| 🔍 **监控数据查询** | 即时查询、范围查询，支持 CPU/内存/磁盘/QPS/错误率/延迟等各类指标 |
| 📊 **交互式图表生成** | 一条命令输出类 Grafana 风格的 HTML 图表，深色主题、鼠标悬停、缩放拖拽、统计卡片一应俱全 |
| 🔔 **告警规则管理** | 创建告警规则、使用预设模板（standard/kubernetes/infrastructure）、规则验证、配置热加载 |
| 📡 **抓取规则配置** | 静态目标配置、文件服务发现、配置验证，支持多种服务发现机制 |
| 🔇 **Alertmanager 静默** | 创建/查看/删除告警静默，维护窗口一键抑制通知 |
| 🔐 **认证支持** | Basic Auth 认证，支持交互式密码输入，401/403 自动提示 |

### 解决什么痛点？

在日常运维中，监控查询是一个**高频但门槛不低**的操作：

1. **PromQL 写不对**：Counter 忘了套 `rate()`、`histogram_quantile()` 聚合丢了 `le` 标签、时间窗口设置不合理……这些都是新手甚至老手常犯的错误。Prometheus Skill 内置了严格的 PromQL 编写规则，AI 在生成查询前会自动校验语法合理性。

2. **API 调用繁琐**：每次查询都要手动拼 curl 命令、处理时间戳、URL 编码……Skill 提供了 Python 脚本工具，一行命令搞定查询 + 图表输出。

3. **告警规则配置易出错**：YAML 缩进、模板变量、severity 分级、for 时间设置……Skill 提供预设模板和规则验证，降低配置出错概率。

4. **跨团队协作难**：不同服务的监控指标散落各处，查询时需要到处找地址和指标名。Skill 通过**标准输入协议**实现引擎与业务解耦，业务团队只需按协议提供参数，查询引擎完全通用。

5. **重复劳动多**：每次排障都要重复"找地址 → 写 PromQL → 调 API → 看结果"的流程。Skill 将这个流程固化为标准步骤，AI 自动执行。

## 💡 使用建议

### 适合什么场景？

Prometheus Skill 支持**三种使用方式**，覆盖从简单到复杂的不同场景：

#### 场景一：快速查询（URL 直接指定）

> 适合：临时排障、个人调试、快速验证

直接在对话中告诉 AI Prometheus 地址和查询意图即可：

```
👤 帮我查一下 http://prometheus.example.com:9090 上过去 6 小时的 QPS 趋势，生成图表
```

AI 会自动构造 PromQL、调用脚本、输出 HTML 交互式图表。**零配置，开箱即用。**

#### 场景二：多 Skill 协作（AI 自主协调）

> 适合：多服务监控、跨模块排障、团队协作

在多 Skill 环境中，业务 Skill 提供指标定义和集群地址，Prometheus Skill 提供查询能力，AI 自动串联：

```
👤 帮我查一下 user-service 的 CPU 使用率

🤖 → 激活业务 Skill → 获取 url + promql → 激活 Prometheus Skill → 执行查询 → 输出结果
```

**新服务接入只需创建业务 Skill 的指标文档，完全不需要修改 Prometheus Skill。**

#### 场景三：自动化流水线（Workflow 编排）

> 适合：日常巡检、批量报告、定时监控

通过 Workflow 将多个步骤编排为自动化流程：

```yaml
steps:
  - name: get_cluster_info      # 获取集群地址
    skill: cluster-info
  - name: query_metrics          # 执行监控查询
    skill: prometheus
    input:
      url: ${{ steps.get_cluster_info.outputs.prometheus_url }}
      promql: 'sum(rate(http_requests_total[5m]))'
      format: chart
  - name: generate_report        # 生成报告
    skill: report-generator
```

**全自动执行，无需人工干预。**

### 如何使用？

1. **安装**：将 Skill 目录复制到项目的 `.cursor/skills/` 或 `.claude/skills/` 下
2. **查询数据**：直接对 AI 说"帮我查 xxx 的 xxx 指标"，提供 Prometheus 地址即可
3. **生成图表**：在查询请求中加上"生成图表"，AI 会自动使用 `--format chart` 输出 HTML
4. **创建告警**：对 AI 说"帮我创建一个 CPU 使用率超过 80% 的告警规则"
5. **配置抓取**：对 AI 说"帮我配置 Prometheus 抓取 app1:8080 和 app2:8080"

> 💡 **小贴士**：`scripts/` 目录下的 Python 脚本仅依赖标准库，**零额外依赖**，任何有 Python 3.6+ 的环境都能直接运行。

## 🧠 创作思路

### 核心设计理念：纯通用引擎 + 标准输入协议

这个 Skill 最核心的设计决策是**三个"不"**：

- ❌ **不内嵌任何业务指标定义**
- ❌ **不内嵌任何业务集群地址**
- ❌ **不感知任何业务上下文**

这意味着 Prometheus Skill 是一个**纯粹的查询引擎**，通过标准输入协议（url + promql）接收参数，不关心参数从哪来。这个设计带来的好处是：

> 无论新增多少服务、变更多少指标，**Prometheus Skill 永远不需要修改**——这就是"开闭原则"在 Skill 体系中的落地。

### 技术巧思

#### 1. Description 写召回，不写百科

```
❌ "Prometheus 是一个开源的监控和告警工具包..."
✅ "支持通过 PromQL 查询监控数据、生成 HTML 交互式监控图表、查看当前活跃告警状态"
```

Description 的目标不是让人看懂，而是让 AI Agent 在面对用户问题时能**精准命中**。所以我把用户会说的关键词（CPU、内存、QPS、错误率、延迟、告警、图表……）全部塞进了 description 和 keywords。

#### 2. 渐进式信息披露

不把所有知识堆在一个文件里，而是分三层：

```
第一层：description → 路由命中（Agent 选对 Skill）
第二层：SKILL.md 正文 → 结构化使用流程（Agent 知道怎么做）
第三层：references/ → 细节知识按需读取（Agent 只读最小必要文档）
```

这样做的好处是**节省上下文窗口**。Agent 不需要一次性消化 5 份 reference 文档，而是根据问题类型只读取 1-2 份。

#### 3. 强制验证流程替代自由推理

监控查询最常见的失败原因是"地址错了"或"PromQL 写错了"。我没有依赖 AI 自己判断，而是在 Skill 中写了一个**强制验证环节**：

```
执行查询前必须验证：
  ├── url 是否已提供？
  ├── promql 是否已提供？
  ├── Counter 类型是否包裹了 rate()/increase()？
  ├── histogram_quantile() 是否保留了 le 标签？
  └── 是否存在无标签过滤的全量查询？
```

把验证写成强制步骤而不是"建议"，能显著提升查询成功率。

#### 4. 负约束比正向知识更重要

Skill 中的 `⛔ 严格规则` 部分，每一条都是从实际踩坑中提炼的：

- Counter 直接查原始值 → 无意义的递增数字
- `rate()` 窗口太小 → 数据点不足导致结果为 0
- 无标签过滤全量查询 → Prometheus OOM
- `histogram_quantile()` 丢 `le` 标签 → 计算结果错误

这些"不要做什么"的规则，对 AI 输出稳定性的提升远大于增加更多背景知识。

#### 5. 七套命令模板降低出错概率

`templates/` 目录提供了 7 套可直接复制使用的命令模板，覆盖从即时查询到告警配置的全部场景。AI 不需要从零构造命令，而是从模板出发做参数替换，大幅降低生成错误命令的概率。

#### 6. 零依赖的 Python 脚本

`scripts/prometheus_query.py` 和 `scripts/prometheus_config.py` 仅使用 Python 标准库，不需要 `pip install` 任何东西。这个决策是刻意的——在运维场景中，目标机器上往往没有完善的包管理环境，零依赖意味着**拿来就能用**。

#### 7. HTML 图表自包含

图表输出基于 ECharts CDN，生成的 HTML 文件是完全自包含的，浏览器直接打开即可。深色主题、交互式操作、统计卡片——不需要 Grafana，一条命令就能得到专业级的监控可视化。

### 设计哲学总结

> **用结构化的规则替代 Agent 的自由推理，用最小必要信息替代全量上下文，用明确的边界替代模糊的能力范围。**

好的 Skill 不是给 AI 追加的一段文档，而是 AI 执行性能的组成部分。Description 决定路由命中率，正文结构决定上下文效率，规则和约束决定回答稳定性。三者缺一不可。
